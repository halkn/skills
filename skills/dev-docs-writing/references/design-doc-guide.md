# Design Doc(設計文書)の書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。実装前に設計とトレードオフの合意を得たい場合に
参照する。「RFC」「Proposal」「Tech Spec」と呼ばれる文書も、目的が実装前の設計合意で
あれば同じ種別として扱う。個別の技術判断1件を短く記録したいだけならADR
(`references/adr-guide.md`)を使う。

## 位置づけ

Design Docは、コーディング開始前に実装戦略と主要な設計判断をトレードオフ重視で記録する、
比較的インフォーマルな文書である。正式仕様書ではない。目的は次の5点にある。

1. 設計問題の早期発見(まだ変更が安価な段階でのレビュー)
2. 組織内合意の形成
3. セキュリティ・可観測性等、横断的関心事の検討を強制すること
4. シニアエンジニアの知見を早い段階で注入すること
5. 将来「なぜこの設計にしたのか」に答えられる記録を残すこと

## 書くかどうかの判断基準

次のいずれかに該当する場合に書く。該当しなければ書かない(文書化コストが合意の価値を
上回るなら省略が正解)。

- 解決策が自明でない
- 複数チーム・複数PRにまたがる
- 後戻りのコストが高い

## 構成

1. **タイトル・メタデータ**: ステータス(Draft/In Review/Approved/Obsolete)、著者、
   レビュアー、最終更新日、関連文書(PRD、親Design Doc、ADR)
2. **概要(Context and Scope)**: 新規読者が前提知識ゼロで読める客観的な背景。
   3〜5段落以内
3. **目標と非目標(Goals / Non-Goals)**: 目標は計測可能に書く(例:「検索P95を
   800msから300msにする」)。非目標は「合理的にあり得るが、あえてやらないこと」を
   列挙する。スコープクリープを防ぐ要となる節
4. **設計(The Actual Design)**: システム構成図(Mermaid等、図を先に・説明を後に)、
   トピック別サブセクション(API・データモデル・処理フロー等)。なぜこの設計に
   したのか(トレードオフ)を必ず書く。実装の細部までは書かない
5. **代替案(Alternatives Considered)**: 検討した各案の利点・欠点・不採用理由。
   最も価値が残る節であり、「他の合理的なエンジニアなら選んだかもしれない案」を
   最低1つは挙げる
6. **横断的関心事(Cross-cutting Concerns)**: セキュリティ・プライバシー・可観測性・
   国際化・ストレージ/コスト見積り。該当なしの場合も「該当なし」と明記し、
   検討したことを示す
7. **マイルストーン・移行計画**: 段階的リリース、既存システムからの移行、
   ロールバック手段
8. **未解決の問題(Open Questions)**: レビューで議論したい論点

## 長さの目安

大規模プロジェクトで10〜20ページ、小規模なら1〜3ページの「ミニDesign Doc」で十分。
長さは対象の複雑さに合わせる。

## 運用ガイド

- レビューはコメントスレッドを基本とする。重要な論点は正式レビューを待たず、
  先にキーパーソンへ直接相談し、開発を止めない。
- Approved後に大きな設計変更が起きたら文書を更新するか、変更点を追補のADRとして
  切り出す。
- アンチパターン: 実装後に清書する「事後Design Doc」(早期発見という主目的を
  失っている)、代替案セクションが空、目標が定性的(「高速化する」のような曖昧語)。

## レビュー時の指摘の出し方

Goals/Non-Goals・代替案・横断的関心事のうち欠けているものを指摘する。目標が
曖昧語(「高速化」「改善」)で書かれている場合は、計測可能な数値目標への書き換えを
促す。代替案が1つも挙がっていない場合は特に指摘する(この節が空だと、将来
「なぜこの設計にしたのか」に答えられなくなる)。
