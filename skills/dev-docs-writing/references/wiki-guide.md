# Wikiページの書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。手順・オンボーディング・FAQなど、繰り返し
参照されるが個別の意思決定記録ではない文書(=Wiki)を新規作成/レビューする際に参照する。

## Wikiが陥りやすい失敗

Google社内Wiki(通称GooWiki)の廃止時、文書の約9割が直近数か月間、閲覧も更新もされて
いなかったことが判明した。原因は次の3つに整理できる。

1. **オーナー不在**: 誰も保守責任を持たないため陳腐化する。
2. **追加プロセス不在**: 既存文書を確認せず新規追加できるため、同じ内容のページが
   複数(実例では同じセットアップ手順が7〜10個)乱立する。
3. **フラットな名前空間**: 階層がないため、重複や陳腐化に気づく手がかりがない。

新規作成時・レビュー時は、この3つの兆候(オーナー不在、既存ページとの重複、
どこにも位置づけられていない孤立ページ)がないかを確認する。

## 単一の真実源(canonical source)

同じ内容を扱うページが複数見つかった場合、そのどれを単一の真実源とするかを決め、
他方は内容を統合するか非推奨(deprecated)であることを明記して残す。「両方をとりあえず
残す」判断は重複を温存し、次にWikiを見た人が再び同じ問題に直面する。

## 冒頭2段落: WHO/WHAT/WHEN/WHERE/WHY

Wikiページは「読者が何を求めて来たか分からないまま読み始める」ことが多いため、
本文に入る前に次を明示する。

- WHO: このページを読むべき対象読者
- WHAT: このページで扱う内容の範囲
- WHEN: 作成日・最終レビュー日
- WHERE: このページが正規の置き場所であるか(他ページからの参照が集まる場所か)
- WHY: このページが必要な理由・背景

多くのエンジニアはHOW(手順)にすぐ飛びつき、WHO/WHAT/WHEN/WHERE/WHYを省略しがちだが、
これらが省略された文書は「Seekers(何が欲しいか分かっている読者)」には冗長に、
「Stumblers(何が欲しいか曖昧な読者)」には手がかり不足になる。

## Azure DevOps Wiki固有の書式

Azure DevOps Wikiに掲載するページは、目次(`[[_TOC_]]`)・図のフェンス
(`::: mermaid` 〜 `:::`)・ページ名の制約など、プラットフォーム固有の規約に従う。
詳細は `references/azure-devops-wiki-guide.md` を参照する。

## レビュー時の指摘の出し方

オーナー・最終レビュー日の有無、他ページとの内容重複の疑い、冒頭のWHO/WHAT/WHEN/WHERE/WHY
の有無を中心に確認する。重複が疑われる場合は、単一の真実源をどちらにするかの判断材料
(最終更新日・参照数など、確認できる範囲のもの)を添えて提案する。
