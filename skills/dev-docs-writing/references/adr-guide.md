# ADR(Architecture Decision Record)の書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。ADRを新規作成/レビューする際に参照する。
実装前の設計全体をこれから合意したい場合はDesign Doc(`references/design-doc-guide.md`)を、
個別の技術判断1件を短く記録したい場合はこちらを使う。

## 原則: 1決定1レコード・追記不可(append-only)

ADRは1つの技術的決定を短く(目安1ページ)記録する。決定が後で変わった場合、
既存のレコードは書き換えずそのまま残し、新しいレコードを書いて相互リンクする。
古いレコードには `Status: Superseded by <新レコードへのリンク>` を追記する。
番号は連番で振り、欠番・再利用はしない。既存ADRの一覧を参照できず次の番号が
確定できない場合は、番号を仮置きにせず要確認事項として扱い、確定作業を依頼者に
委ねる。

この運用により、「なぜ当時その決定をしたのか」という記録が後から上書きされずに残る。
レビュー時、既存レコードを直接編集して結論を差し替えているものを見たら、append-onlyの
原則に反しているため、新レコード+superseded の形に直すよう指摘する。

## 構成

1. **タイトル**: 決定の内容が一目でわかる名詞句(「なぜ〜か」ではなく「何を決めたか」)
2. **Status**: Proposed / Accepted / Superseded のいずれか。依頼文が「〜した」のように
   既に実施済みの決定を報告している場合はAccepted、これから承認を得る段階であれば
   Proposedとする。
3. **Context**: なぜこの決定が必要になったか。制約・前提条件。
4. **Decision**: 何を決めたか。断定的に書く。
5. **Consequences**: 決定によって生じる結果(良い面・悪い面の両方)。
6. **却下した代替案**: 検討したが採用しなかった選択肢と、却下した理由。

「却下した代替案」は省略されがちだが、長期的に最も価値が高い節である。将来同じ
チームが同じ検討を繰り返すのを防ぐため、新規作成時は必ず埋めるよう促す。

## よくある失敗

- 1つのADRに複数の決定を詰め込む → 分割を提案する(目安: 1ページを超えたら複数決定が
  混在している兆候)。
- 決定が変わった際に本文を書き換える → 新レコード+superseded に直す。
- Consequencesに良い面しか書かない → トレードオフとして悪い面も明記させる。
- 却下した代替案の理由が書かれていない → 「なぜ採用しなかったか」を一文でも追加させる。

## レビュー時の指摘の出し方

既存ADRをレビューする場合、上記の構成要素(Status/Context/Decision/Consequences/却下した
代替案)のうち欠けているものを列挙し、該当箇所を引用しながら改善案を提示する。決定内容
そのものの当否には踏み込まず、記録としての体裁を対象にする。
