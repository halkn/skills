# READMEの書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。READMEを新規作成/レビューする際に参照する。

## 原則: READMEはプロジェクトの入口であり、網羅的ドキュメントそのものではない

READMEの役割は、初めてこのプロジェクトに来た人に第一印象を与え、次にどこを見れば
よいかを示すことにある。手順の詳細・設計判断の経緯・運用手順を全てREADMEに書き込むと、
本来の役割である「入口」としての読みやすさが失われる。詳細情報はWiki・設計ドキュメント・
Runbookなど、目的に応じた他の文書種別に切り出し、READMEからはリンクする。

## 原則: 内容の型を混在させない

READMEの中身は、性質の異なる情報(概念の説明・手順・仕様・トラブルシューティング)が
混在しやすい。「これは説明なのか、手順なのか、仕様なのか」を意識して節を分け、1つの
節に異なる型の情報を混ぜない。手順の節に設計判断の説明を混ぜると、実行したい読者が
読み飛ばせなくなる。

## 構成

1. 目的・概要: このプロジェクトが何をするものか、一段落程度で
2. 前提条件: 必要なランタイム・ツールをバージョン付きで明記する(例:「Node.js 22以上」)
3. セットアップ: 環境構築・依存関係のインストール手順。コピー&ペーストで完了する
   最短手順を番号付きリストで書き、各コマンドの後に期待される結果(例:
   「`http://localhost:3000` で起動を確認する」)を添える
4. 使い方: 基本的な実行方法・代表的なコマンド
5. 設定: 主要な設定項目(詳細は設定リファレンスにリンクしてもよい)
6. 貢献方法へのリンク: 開発フロー・コーディング規約等がある場所へのリンク

## 執筆時の注意

- コマンド例では、行頭に `$` のようなプロンプト記号を付けない。読者がそのままコピー
  して実行できることを優先する。
- セットアップ手順は、実際に新しい環境で再現できるかを最重視する。社内VPN・環境変数・
  権限申請のような暗黙の前提を書き漏らさない。

## レビュー時の指摘の出し方

上記6項目のうち欠けているものを指摘する。逆に、設計判断の詳細やインシデント対応手順
など、他の文書種別が担うべき内容がREADMEに書き込まれている場合は、切り出し先の候補
(設計ドキュメント・Runbook等)を示した上で分割を提案する。手順の節に説明的な内容が
混ざっている場合も、型の混在として指摘する。
