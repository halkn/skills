# オンボーディング資料/ハンドブックの書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。新メンバー向けの参加手順やチームの働き方を
新規作成/レビューする際に参照する。

## 原則: handbook-first

口頭やチャットで伝えた知識は消える。プロセス・規約・暗黙知はハンドブックに書き、
質問には回答そのものではなくリンクで答える。答えとなる文書が存在しない場合は、
まず文書を更新してからそのリンクを共有する。

## 原則: 読む資料ではなく実行するチェックリストとして設計する

オンボーディング資料はチェックリスト形式にすると、完了したかどうかを検証できる。
各項目に「完了の定義」を添える(例:「環境構築完了 = ローカルでテストが全緑」)。

## 構成

1. 対象(入社/異動してきた開発メンバー等)と目標(例:「N日目までに最初のPRを
   マージする」)
2. Day 1チェックリスト: アカウント・権限申請(承認者・所要日数付き)、開発環境構築
   (README等の実体へのリンク。手順をここに複製しない)、必読文書へのリンク
3. Week 1チェックリスト: 最初の小さなタスクの完了、定例・オンコール体制・
   コミュニケーション規約の把握
4. チームの働き方: 開発フロー(ブランチ戦略・レビュー規約・CI/CD)、コミュニケーション
   (同期/非同期の使い分け、意思決定の残し方)、主要システムの一覧(オーナーと
   Runbook/Design Docへのリンク)
5. 用語集: 社内略語・プロダクト固有語。新メンバーが聞き返した語は必ず追加する

## 運用ガイド

- 手順の実体はREADME・Runbook等の単一の真実源に置き、オンボーディング資料は
  リンク集と順序付けに徹する。手順を複製すると、複製元が変わった瞬間に陳腐化する。
- 新メンバーは最も厳しい文書テスターである。詰まった箇所はその場で文書を修正して
  もらうとよい(最初のPRがドキュメント修正であることは良い兆候である)。
- 30/60/90日の期待値を明文化すると、本人とマネージャー双方の認識合わせになる。
- 更新責任: 直近でオンボーディングされた人が改善のPRを出す運用にすると、
  最も鮮度の高い視点で更新され続ける。

## レビュー時の指摘の出し方

手順が他文書(README・Runbook等)と重複して書かれていないか、各チェックリスト項目に
完了の定義があるか、リンク切れがないかを中心に確認する。「気をつける」「周知する」の
ような具体タスクを伴わない項目は、実行可能な粒度への書き換えを提案する。
