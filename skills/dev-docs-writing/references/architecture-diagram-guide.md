# アーキテクチャ図・データフロー図の書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。システム構成図・データフロー図(パイプライン設計書に
付随する図を含む)を新規作成/レビューする際に参照する。

## 原則: diagram as code

図はリポジトリ内にMermaid等のテキスト形式(diagram as code)で保持し、コードと同じ
PRでレビュー・バージョン管理する。画像ファイルとして別管理すると、コード変更に
追随せず陳腐化しやすい。図をAzure DevOps Wikiページに直接掲載する場合は、標準の
Mermaidフェンスではなく `references/azure-devops-wiki-guide.md` のフェンス記法に
置き換える。

Mermaid記法そのもの(構文エラーの回避・GitHub/GitLab等での描画互換性・大規模図の
レイアウト)については[mermaid-diagrams](../../mermaid-diagrams/SKILL.md)を参照する。

## C4モデル

C4モデル(Context / Container / Component / Code の4階層)を使うと、詳細度の異なる
図を一貫した語彙で描き分けられる。

1. **Context**: このシステムが、どの利用者・外部システムとやり取りするか
2. **Container**: システムを構成する実行単位(サービス・データストア等)とその関係
3. **Component**: Container内部の主要なモジュール構成
4. **Code**: クラス図等、実装レベルの詳細

多くのチームには Context 図と Container 図の2つで十分であり、Component/Code まで
描くかどうかは対象の複雑さと更新コストに応じて判断する。最初から4階層すべてを
要求しない。

## データフロー図・パイプライン設計書への適用

データの入出力・依存関係を示す図(パイプラインのDAG等)も、C4のContainer図に相当する
粒度で捉えられることが多い。各ノードの入力・出力・依存関係が分かる形で描く。

## レビュー時の指摘の出し方

図が画像ファイルとしてのみ存在し、テキスト形式(diagram as code)で保持されていない
場合はその旨を指摘する。詳細度が対象の複雑さに対して過剰(不要にComponent/Codeまで
描いている)、または不足(Contextすら無い)していないかを確認する。
