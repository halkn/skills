# Runbookの書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。Runbookを新規作成/レビューする際に参照する。

## 原則: 1アラート1Runbook

1つのRunbookは1つのアラート・1つのインシデントシナリオだけを扱う。複数のシナリオを
1つのRunbookに詰め込むと、深夜に対応するオンコール担当者や新任のエンジニアが必要な
手順にたどり着けなくなる。既存のRunbookが複数シナリオを扱い始めていたら、内容が
80%重複していても分割を提案する(重複は、シナリオが混在した1つの巨大なRunbookより
実害が小さい)。

## 原則: 判断を減らす

想定読者は「深夜に呼ばれた、このシステムに詳しくないオンコール担当者」である。この
読者像で全記述を検査する。「状況に応じて対応する」という記述はRunbookとして失敗して
いる。分岐が必要な箇所は、条件を明示したif-then形式で書く(例:「エラー率が5%を超えて
いれば手順Aへ、それ以外は手順Bへ」)。読者が自分で判断しなければならない箇所を残さない。

## 構成

1. **対象の一行定義**: 例「Payments DB — connection pool exhaustion (Sev 1-2)」
2. **エスカレーション先**: 二次オンコール・チームリード等、連絡手段付きで明記する
3. **前提・必要権限**: 手順の実行に必要なアクセス権限(本番DB読み取り権限等)と、
   未取得の場合の取得方法へのリンク
4. **Symptom(症状)**: アラート名、エラーメッセージ、逸脱しているメトリクス、ユーザー影響
5. **Diagnosis(診断)**: 何を確認するかだけでなく、どう確認するか(具体的なコマンド・
   ダッシュボードへのリンク)まで書く。分岐がある場合はif-then形式にする
6. **Mitigation(緩和)**: 応急処置。副作用のあるコマンド(再起動・強制終了・データ削除等)は
   実行前に必ず影響範囲を明記した警告ブロックを添える
7. **復旧確認**: 正常に復旧したと判定できる具体的な条件(メトリクス値・確認URL)
8. **ロールバック/中断手順**: 手順の途中で中断する場合の安全な戻し方
9. **Permanent fix(恒久対策)**: 根本原因への対応。既存の対応で終わっている場合は
   「未実施」であることも明記する

この順序(症状→診断→緩和→復旧確認→恒久対策)を入れ替えない。診断より先に緩和策を
書くと、オンコール担当者が原因を特定しないまま対症療法を実行してしまう危険がある。

## リンクと運用上の注意

- ダッシュボード・スクリプト・関連コマンドへのリンクを本文中に埋め込む。
- アラート設定から該当Runbookへ直接リンクする運用を前提に書く。検索性の悪いWiki階層に
  埋もれたRunbookは、障害発生時には実質的に存在しないのと同じになる。
- 実際に使われた・訓練されたことのないRunbookは信頼できない。使われた形跡がない
  Runbookをレビューする場合、その旨を指摘に含めてよい。
- 古いRunbookは「無い」より悪い場合がある(誤った手順で状況を悪化させうる)。長期間
  更新されていないRunbookは、内容の陳腐化を疑いオーナーに確認を促す。
- 実行頻度が高いRunbookは自動化の候補としてマークする。手順書の成熟の最終形は
  自動化であり、頻繁に手動実行されている手順は自動化を検討する価値がある。

## レビュー時の指摘の出し方

症状→診断→緩和→復旧確認→恒久対策の順序が保たれているか、副作用のあるコマンドに
警告が付いているか、「状況に応じて対応する」のような判断を読者に委ねる記述が
残っていないか、オーナー・最終レビュー日があるかを中心に確認し、欠けている要素を
該当箇所付きで指摘する。
