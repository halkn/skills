# レビューチェックリスト

`dev-docs-writing/SKILL.md` の詳細ガイド。新規作成時のセルフチェック、既存文書レビュー時の
双方から参照する。上から順にチェックし、欠けている項目を該当箇所付きで指摘する。

## 1. 共通(全文書種別)

- [ ] オーナーが明記されている(Wiki・Design Doc・ADR・Runbook・オンボーディング/ハンドブック)
- [ ] 最終レビュー日が明記されている(Wiki・Design Doc・ADR・Runbook・オンボーディング/ハンドブック)
- [ ] 最終レビュー日から長期間(目安: 半年以上)経過しており、内容が陳腐化していないか
- [ ] 1文書1目的が守られている(複数の目的が混在していないか)
- [ ] コードや設定から自動的にわかる情報(全カラムの列挙、パラメータの網羅等)を
      重複して書き写していないか。書き写しは参照リンクへの置き換えや削除を提案する

## 2. 種別固有の構成

### Wiki

- [ ] 冒頭2段落でWHO/WHAT/WHEN/WHERE/WHYが示されている(`wiki-guide.md`)
- [ ] 既存の他ページと内容が重複していないか

### Design Doc

- [ ] Status(Draft/In Review/Approved/Obsolete)が明記されている
- [ ] Goals/Non-Goalsが計測可能に書かれている(曖昧語で済ませていないか)
- [ ] 代替案(Alternatives Considered)が1つ以上あり、利点・欠点・不採用理由が揃っている
- [ ] 横断的関心事(セキュリティ・可観測性等)に触れているか(該当なしの明記でもよい)

### ADR

- [ ] Status(Proposed/Accepted/Superseded)が明記されている
- [ ] Context/Decision/Consequencesが揃っている
- [ ] 却下した代替案とその理由が書かれている
- [ ] 決定が変わった場合、本文を書き換えず新レコード+supersededの形になっているか

### Runbook

- [ ] 対象アラート・シナリオが一行で特定できる
- [ ] エスカレーション先・前提権限が明記されている
- [ ] Symptom→Diagnosis→Mitigation→復旧確認→Permanent fixの順序になっている
- [ ] 副作用のあるコマンドに警告が添えられている
- [ ] 「状況に応じて対応する」のような読者に判断を委ねる記述が残っていないか
- [ ] 複数シナリオが1つのRunbookに混在していないか

### README

- [ ] 目的・概要/前提条件/セットアップ/使い方/設定/貢献方法へのリンク、の6項目が揃っている
- [ ] コマンド例に期待結果が添えられているか、`$`のようなプロンプト記号が付いていないか
- [ ] 他の文書種別が担うべき詳細(設計判断・インシデント対応手順等)が混入していないか

### アーキテクチャ図・データフロー図

- [ ] diagram as code(Mermaid等のテキスト形式)で保持されているか
- [ ] 対象の複雑さに対して詳細度(C4のどの階層まで描くか)が適切か

### APIドキュメント

- [ ] 設計判断や実装の経緯が混入せず、仕様(振る舞い)のみが書かれているか
- [ ] パラメータ表に制約(範囲・フォーマット・デフォルト値)があるか
- [ ] レスポンス例が実装と乖離していないか(ダミー型名で済ませていないか)
- [ ] エラー時のクライアント側の対処(リトライ可否等)が書かれているか

### オンボーディング/ハンドブック

- [ ] チェックリストの各項目に完了の定義があるか
- [ ] 手順の実体が他文書(README・Runbook等)と重複していないか
- [ ] 「気をつける」「周知する」のような具体タスクを伴わない項目がないか

## 3. 文体(`ja-writing-style.md`)

- [ ] 説明部/仕様部の区別に応じたルールが適用されているか
- [ ] 一段落一トピックになっているか(説明部)
- [ ] 推量表現を根拠なく断定に変えていないか
- [ ] 受動態でなく行為者主体で書かれているか
- [ ] LLM的表現(予告・総括型、姿勢宣言、空虚な形容・動詞、根拠のない緩和、接続の型)が
      使われていないか
- [ ] 同じ主張の繰り返しや、描写直後の要約がないか
- [ ] 見出しが具体的で、内容を特定できるか

## 指摘の出し方

- 該当箇所を引用してから、問題点と改善案をセットで示す。
- 指示なく無断で全文を書き換えない。改善案は提案として示し、適用は依頼者の判断に委ねる。
- Wikiで重複が疑われる場合は、単一の真実源の候補と、統合または非推奨化の進め方を添える。
- 削除も改善である。陳腐化・重複した記述に気づいたら、追記の提案だけでなく削除や
  参照リンクへの置き換えも選択肢として示す。

## 指摘の重大度

指摘には次のいずれかのラベルを付ける。

| ラベル | 意味 |
|---|---|
| **[Must]** | 誤り・誤解を招く記述・重大な欠落。修正が必須 |
| **[Want]** | 読者体験を大きく改善する提案。対応を推奨 |
| **[Nit]** | 表記・文体の細かい指摘 |

個別のチェック項目がどのラベルに当たるか迷う場合は、「将来の意思決定や読者の解釈を
誤らせる欠落か」を基準にする。誤らせるなら[Must](例: 代替案が皆無、目標が非計測的)、
読みやすさや運用上の利便性の話にとどまるなら[Want]とする。

レビューの目的は完璧な文書ではなく、改善の速いサイクルである(Good Over Perfect)。
[Nit]のみを理由に差し戻しを求めない。文書を今より良くする指摘をまず示し、
残りの改善は次回の更新に回してよい、という温度感で指摘量を調整する。
