# 日本語文体ルール

`dev-docs-writing/SKILL.md` の詳細ガイド。開発チーム内のWiki・設計ドキュメント・Runbook・
README・アーキテクチャ図の説明文を書く/レビューする際の文体ルール。SKILL.md本体には
最重要項目(LLM的表現の禁止リスト)のみを置き、詳細と具体例をここに集約する。

## 適用範囲の切り替え: 説明部と仕様部

文書には**説明部**(背景・設計判断・代替案の理由・根本原因の分析)と**仕様部**
(パラメータ表・手順・チェックリスト)がある。「段落・論証構成」「論証の厳密さ」の
規範は説明部にのみ適用する。仕様部はリスト・表で走査可能に書くことを優先し、
「整形」「表記の機械的ルール」「視点・語り」「LLM的表現の禁止」「読者への誠実さ」
のみ適用する。手順の1ステップを段落構成の規範に従わせようとしない。

## 整形

- 一文ごとに改行し、段落は空行で区切る。
- コード・ログ・コマンドはコードブロックに入れ、地の文に埋め込まない。
- 補足情報は本文中に差し込まず、脚注または別セクションに降ろす。
- 定義した用語は太字で示し、以降その用語に言及するときは「」で囲む。
- ダッシュ(—)や中黒(・を除く記号的な区切り)を地の文の区切りとして使わない。
- 見出しは単一の自然な句にする。「AとB」のように区切り線で二要素を無理に詰め込まない。

## 表記の機械的ルール

- 英数字は半角を使う(全角英数字は使わない)。
- 和文と半角英数字の間には半角スペースを入れる(例:「dbtモデルを 3 つ作成する」)。
  ただし括弧・句読点との間にはスペースを入れない。
- カタカナ語の長音符は原則として付ける(サーバー、ユーザー、パラメーター)。
  3音以下の慣用語(メモリ、ディレクトリ等)は長音なしも許容するが、同一文書内では
  表記を統一する。
- テーブル名・カラム名・ファイルパス・コマンド・環境変数・識別子はバッククォートで
  囲む(例: `stg_orders`、`profiles.yml`)。

## 段落・論証構成

- 一段落は一トピックに絞る。
- 段落の先頭文で、その段落が何を述べるかを示す。
- 前の段落との関係(対比・補足・帰結)を接続表現で示す。
- 結論は一度だけ述べる。誤った解釈がありうる場合は、それを明示的に否定してから
  正しい理解を述べる。
- 最も重要な情報を、関係のない前置きの奥に埋めない。

## 論証の厳密さ

- 推量・可能性を示す表現(「〜の可能性がある」等)を、根拠なく削って断定に変えない。
- 性質の異なるものを同一視しない。複数の要因を単一の原因に還元しない。
- 因果関係を述べるときは、結果に至る機構(なぜそうなるか)まで説明する。
- 常に成り立つわけではないことを「必ず」できるかのように書かない。条件を明示する。
- 本文中で触れた懸念や疑問(伏線)は、後の段落で必ず答えを示す。
- 複数の概念をまとめて上位語で呼ぶ場合、なぜその括りが妥当かを一文で示す。

**悪い例**: 「このRunbookに従えば必ず復旧します。」
**良い例**: 「このRunbookの手順は、connection pool exhaustionが原因の場合に有効です。
別の原因(ネットワーク分断等)では復旧しないことがあります。」

## 読者負荷管理

- 後で参照しない固有名詞(一時的なファイル名等)を本文に出さない。
- 固有名詞を出す場合は、同格の説明を添えてその場で特定できるようにする。
- 新しい例を追加するときは、既存の例との違いと、なぜ追加が必要かを前置きする。
- 節の導入部分に、その節の主題と無関係な詳細を書かない。

## 視点・語り

- 受動態ではなく、行為者を主語にした能動的な文で書く。
- 「あなた」のような二人称で読者を呼ばず、役割名(オンコール担当者・レビュアー等)で呼ぶ。
- 対象を「システム」「ツール」のような曖昧な語で済まさず、具体的な名前で書く。
- 一度導入した用語を、以降で曖昧な言い換えに後退させない。
- 慣用的な言い回しは、そのチーム・分野で定着している語を選ぶ。

## 演出の抑制

- 修辞的な表現は、実際に効果が生じる箇所のみで使う。
- 短い決め台詞だけの独立した段落を多用しない。
- 太字による強調は、論理的に要となる箇所に文書1つあたり1〜2箇所程度にとどめる。
- 命令調(「〜せよ」)よりも、判断を促す言い方(「〜を確認する」)で書く。
- 話の転換点を過剰に劇的な言い回しで演出しない。
- 「重要なのは」のような、内容を伴わない予告的な前置きを避ける。
- 一意に解釈できない比喩やひねった言い回しを使わない。

## LLM的表現の禁止(詳細)

SKILL.md本体に挙げた5項目それぞれの具体例。

| 分類 | 悪い例 | 理由 |
|---|---|---|
| 予告・総括型 | 「本節ではRunbookの構成を扱う。」「以上、ADRの書き方を見てきた。」 | 内容を先取り・反復するだけで情報が増えない |
| 姿勢宣言 | 「この問題に正面から向き合う。」 | 何をするかを述べておらず読者の判断材料にならない |
| 空虚な形容 | 「オーナーの明記は不可欠だ。」 | 「なぜ不可欠か」を言わないまま評価語だけを置いている |
| 空虚な動詞 | 「陳腐化の背景を掘り下げる。」 | 具体的に何を確認するのかが特定できない |
| 根拠のない緩和 | 「重複していると言えるだろう。」(重複が明白な場合) | 断定できる内容をぼかしている |
| 接続の型 | 「運用の観点において、オーナーが必要となる。」「さらに、また、加えて」の連打 | 「〜において」「〜という観点から」は新しい情報を運ばず、接続語の連打は論理関係を示さない |

このほか、「非常に」「極めて」のような中身のない強調、「基本的に」「適宜」「なるべく」
「等」の乱用(条件・範囲を明示せず済ませる)、「簡単に」「すぐに」(読者のスキルに
依存し人によって意味が変わる)、「一旦」「とりあえず」(暫定であることを示すだけで
期限や見直し条件を伴わない)も、根拠や具体性を伴わない場合は避ける。

## 冗長排除

- 同じ主張を言い方を変えて繰り返さない。
- 隣接する節で同じ内容が重複していたら、一方に統合する。
- 描写した直後に、その内容をそのまま要約し直さない。
- 読者が文脈から補える中間段階の説明を書かない。
- 接続や評価だけを述べて情報を追加しない文を置かない(例:「これは興味深い」)。
- 想像上の読者との問答を修辞として使わない(例:「では、なぜそうなるのか。」)。
- 「本ガイドもそれを否定するものではない」のような立場弁明を書かない。

## 見出し付け

- 見出しは、その節の内容を特定できる具体的な語句にする。
- 作業手順だけでなく、扱う対象や問いを見出しに反映する。
- 疑問形にするか名詞句にするかは、文書全体のトーンに合わせて統一する。

**結論を見出しに含めるかどうかはトレードオフ**: 元の規範は「見出しで結論を完全に
言い切らない」だが、走査読みされるWiki・レビュー観点の設計ドキュメントでは、結論を
含む見出し(例:「セッションストアにRedisを採用する」)の方が目的の情報に素早く
たどり着ける。ADRのタイトルは常に決定内容を言い切る結論形にする。一方、論証を順に
読ませたい節(設計上のトレードオフの検討過程等)では、結論見出しは論証を先取りして
読む動機を失わせるため避ける。文書の性質に応じて判断する。

## 読者への誠実さ

- 作為的に見える例(都合の良すぎる数値等)は隠さず、根拠を添えて示す。
- 「筆者はこう考える」ではなく、確認できる一般的事実に基づいて書く。
- 未確認の事項を、確認済みであるかのように書かない(「〜と思われる」等で明示する)。

## 原典から取り込まなかった規範

このルールは、日本語文章規範([k16shikano「japanese-tech-writing」](https://gist.github.com/k16shikano/fd287c3133457c4fd8f5601d34aa817d))を
開発チーム内文書向けに再構成したものである。次の項目は技術書籍固有の規範と判断し、
取り込んでいない。

- 山場の演出・溜め・決め台詞・感嘆符の制御(演出の抑制の大半)
- 作中人物の視点・架空の人物設定・物語的な場面進行
- 脚注記法・コラム記法(開発チーム内文書ではリンクや別セクションに置き換える)
- 「山場で効かせる情報の先出し禁止」のような物語構成の規範
