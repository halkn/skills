# APIドキュメントの書き方

`dev-docs-writing/SKILL.md` の詳細ガイド。APIのエンドポイント仕様・使い方を
新規作成/レビューする際に参照する。

## 原則: 仕様と実装判断を分離する

APIドキュメントはリファレンス(仕様)であり、設計判断や実装の詳細を書く場所ではない。
「なぜこのAPI設計にしたか」はDesign DocやADRに書き、APIドキュメントには「今この
APIはどう振る舞うか」だけを書く。インターフェースと実装を分離するのと同じ考え方を
ドキュメントでも守る。

可能な限り、コードに近い場所(OpenAPI定義・docstring等)を単一の真実源とし、手書きの
Markdownはその生成元か補完に限定する。仕様とドキュメントの二重管理は陳腐化する。

## 読者

読者は「今すぐこのAPIを呼びたい開発者」である。説明より先に、動く例を優先する。

## 文書全体の構成

文書冒頭にOwner・最終レビュー日を置き、以降はエンドポイントごとに次の構成を
繰り返す。

## エンドポイントごとの構成

1. **タイトル**: 動詞句(例: 「ユーザーを作成する」)
2. **概要**: 何をするAPIか、どんなときに使うかを1文で
3. **エンドポイント**: `POST /api/v1/users` のようなメソッドとパス
4. **必要な権限・スコープ**
5. **リクエスト**: パラメータ名・型・必須かどうか・説明と制約(範囲・フォーマット・
   デフォルト値)を表で
6. **レスポンス**: 成功時のJSON例。実際に動く値を使い、`"string"` のような
   ダミー型名を書かない
7. **エラー**: ステータスコード・発生条件・エラーコードに加えて、クライアントが
   どう対処すべきか(リトライ可否・バックオフ)まで書く
8. **例**: コピー&ペーストで動くcurlの完全な例と、主要言語1つのコード例
9. **備考**: レート制限、冪等性、ページネーション、非推奨予定

## バージョニング・非推奨化

導入バージョン・Deprecated表示・廃止予定日・移行先へのリンクを明示的に書く。
非推奨になった機能も、実際に削除するまではドキュメントに残し、その旨を告知する。

## レビュー時の指摘の出し方

設計判断や実装の経緯がAPIドキュメントに書き込まれている場合は、Design DocやADRへの
切り出しを提案する。パラメータ表に制約(範囲・デフォルト値)が欠けている、レスポンス例が
実装と乖離していそうな記述、エラー時のクライアント対応が書かれていない、といった点を
中心に指摘する。
