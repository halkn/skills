# データ基盤アーキテクチャのアンチパターン集

設計の自己チェックとレビューでの照合に使う。各項目は「症状 / なぜ起きるか / 対策」の
形式。確信度の注記がある項目は、公式ドキュメント以外(ベンダー・コンサルの知見)に
由来するため、環境依存の可能性を踏まえて適用する。

## コンピュート・性能のアンチパターン

### 単一巨大ウェアハウスへの全ワークロード集約

- 症状: ETL・BI・アドホックが同一の大型コンピュートを共有している。
- なぜ起きるか: 管理対象を減らしたい、初期構築の簡便さ。
- 対策: ワークロード別にコンピュートを分離する。分離しないとサイジング・キャッシュ・
  課金追跡のすべてが最適化不能になる。Snowflake公式Performanceガイドも
  明示的にアンチパターンとしている。

### オーバーサイジング(「念のため大きめ」)

- 症状: 根拠なくXL以上で開始。サイズ選定の測定記録がない。
- なぜ起きるか: 性能問題への不安。サイズを下げる意思決定は上げるより心理的に難しい。
- 対策: start small, scale as needed。サイズを1段上げるとコストは概ね倍になる。
  性能目標(SLO)を先に定め、測定してから上げる。

### スケールアウトで遅いクエリを直そうとする

- 症状: 単体クエリが遅いのでマルチクラスタ(並列数)を増やしている。
- なぜ起きるか: スケールアップ(単体性能)とスケールアウト(同時実行)の混同。
- 対策: 遅いクエリはサイズアップかクエリ・データレイアウト改善。マルチクラスタは
  キュー待ち(同時実行の詰まり)対策であり、単体クエリは1クラスタでしか実行されない。

### 自動停止(auto-suspend)の未設定・過長設定

- 症状: アイドル状態のコンピュートが課金され続けている。
- なぜ起きるか: デフォルト設定のまま放置、キャッシュ温存の過大評価。
- 対策: ETL系は即時〜1分、BI系はキャッシュ効果と天秤にかけて5〜10分程度を起点にする。
  自動再開は全ウェアハウスで有効化する(起動は数秒)。

## SQL・データモデルのアンチパターン

### SELECT *

- 症状: パイプライン・ビュー定義に `SELECT *` が常用されている。
- なぜ起きるか: 記述の手軽さ。上流スキーマ変更への「柔軟さ」という誤解。
- 対策: 列を明示する。カラムナストレージの列プルーニングが効かず、
  スキーマ変更を無検知で下流に伝播させる。静的解析で最頻出のアンチパターン。

### JOIN条件の欠落・不完全(直積の発生)

- 症状: 行数が掛け算で爆発し、コストと実行時間が急増する。
- なぜ起きるか: 複合キーの一部だけでJOINする、テストデータでは顕在化しない。
- 対策: JOINキーのユニーク性をテスト(dbt tests等)で担保する。
  行数の入出力比を監視する。

### mixed grain(粒度混在)のマート

- 症状: 日次ファクトと月次集約が同一テーブルに同居。ダッシュボードで重複カウント。
- なぜ起きるか: 「1テーブルで済ませたい」という利便性優先。
- 対策: 1テーブル1粒度。集約レベルが異なるならテーブルを分ける。

### オンプレOLAP設計の直移植

- 症状: 過度に正規化されたスキーマや、インデックス前提の設計をそのまま移植。
- なぜ起きるか: 移行プロジェクトの時間圧力、既存資産への愛着。
- 対策: カラムナ・マイクロパーティション前提でスキーマをレビューしてから移行する。
  クラウドDWHではJOIN削減・ワイドテーブルが有利な場面が多い。

## アーキテクチャ全体のアンチパターン

### 統制外データレイクの併設

- 症状: DWHの外に「とりあえず置く」オブジェクトストレージ層があり、
  アクセス制御・カタログ・品質管理を二重に実装している。
- なぜ起きるか: 「将来使うかもしれない」データの抱え込み。
- 対策: ガバナンスは一元化する。プラットフォーム内で管理できるものを外に出さない。
  外部テーブル・オープンフォーマット(Iceberg等)を使う場合もカタログと統制は統合する。

### セマンティック層・メトリクス定義の欠如

- 症状: 同じKPIがダッシュボードごとに異なる値を示す(KPIドリフト)。
- なぜ起きるか: メトリクス定義が各BIツール・各クエリに散在。
- 対策: メトリクス定義を単一の場所(セマンティック層・dbt metrics等)に集約し、
  Git管理とCI検証の対象にする。

### リアルタイム要件へのバッチ+リバースETLの多段構成

- 症状: 鮮度SLOが分単位なのに、バッチ集計→リバースETLの多段でレイテンシが積み上がる。
- なぜ起きるか: 既存バッチ資産の再利用を優先。
- 対策: 鮮度SLOから逆算して経路を設計する。ストリーミング取り込みや宣言的な
  増分パイプラインを検討する。(確信度: 中。ストリーミング推進ベンダー由来の見解)

## ガバナンス・組織のアンチパターン

### アクセス制御・マスキングの後回し

- 症状: 「データを入れてから権限を整える」計画になっている。
- なぜ起きるか: 初期はデータ投入の進捗が優先されがち。
- 対策: データ着地・アクセス付与後のRBAC再設計は極めて困難。ロール階層・
  マスキング方針は設計時に作り込む。

### 管理者権限の日常利用

- 症状: 最上位権限(ACCOUNTADMIN相当)がデフォルトロールや自動化スクリプトに使われている。
- なぜ起きるか: 「権限エラーが出ないので楽」。
- 対策: 管理者権限は管理作業専用にし、多要素認証と監査を必須にする。
  自動化にはサービス用の限定ロールを払い出す。

### オンプレ思考のままの移行(人の問題)

- 症状: 従量課金への意識がない使い方、手作業運用の温存、ネイティブ機能で足りる
  ガバナンスの過剰な自作。
- なぜ起きるか: ツールだけ移行して運用思考が移行していない。
- 対策: アップスキル計画を移行計画に含める。チームスキル適合性の観点で
  設計をレビューする([review-lens.md](review-lens.md))。
