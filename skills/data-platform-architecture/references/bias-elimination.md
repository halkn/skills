# バイアス排除検証プロトコル

設計成果物を確定する前に、設計者(このスキルを実行しているClaude自身を含む)の
認知バイアスを構造的に排除して品質を検証するための手順。3つの層からなり、
層1・層2は設計中に常時、層3は成果物確定前に1回以上実施する。

## 排除対象のバイアスと構造的対策

「気をつける」だけではバイアスは排除できない。各バイアスに、それを機械的に
無効化する構造的な対策を対応させる。

| バイアス | 設計での現れ方 | 構造的対策 |
|---|---|---|
| 確証バイアス | 自分の構成案を支持する根拠ばかり集め、反対材料を軽視する | 層3: 設計の経緯を知らない独立レビュアーによる検証 |
| アンカリング | 最初に思いついた構成に固執し、以降の検討がその微修正に留まる | 層2: 主要判断ごとに代替案を最低1つ挙げ、却下理由を明文化する |
| 新技術・流行バイアス | 要件にない新機能・流行ツールを「せっかくなので」採用する | 「必要なだけ速く」原則との照合、チームスキル適合性の評価([review-lens.md](review-lens.md)) |
| 権威バイアス | 「公式が推奨」「有名企業の事例」を文脈を確認せず適用する | 根拠に確信度ラベルを付ける(下記) |
| サンクコスト | 指摘を受けても「ここまで作ったから」と設計を変えない | 層3の指摘すべてに対応/見送りを理由付きで記録し、先送りを許さない |
| 利用可能性ヒューリスティック | 直近に見た事例・自分が詳しい領域に検討が偏る | 層1: チェックリストで全観点を強制的に走査する |

**根拠の確信度ラベル**: 設計書の主要判断には根拠の出所を次の3段階で明記する。

- `[公式]` プラットフォームの公式ドキュメント・公式ガイド由来
- `[ベンダー/コンサル]` 最適化ツールベンダー・コンサルのブログ等由来(環境依存の可能性)
- `[一般論]` 業界の通説・経験則(定量的裏付けなし)

## 層1: チェックリスト自己検証

設計中、次の2ファイルを開いて全項目と照合する。「該当なし」も判断として扱い、
飛ばした観点を残さない。

- [review-lens.md](review-lens.md) — 5柱+チームスキル適合性の観点
- [antipatterns.md](antipatterns.md) — 既知の落とし穴

利用可能性ヒューリスティック(検討の偏り)への対策なので、自信のある領域ほど
省略しないこと。

## 層2: 反証の明文化

主要な設計判断(プラットフォーム選定・コンピュート構成・レイヤリング・
パイプライン方式・アクセス制御方式など)ごとに、設計書の「主要設計判断」
セクションに次を必ず記載する。書けない判断は検討が不足している。

- **判断**: 何をどう決めたか
- **根拠**: なぜそう決めたか(確信度ラベル付き)
- **失敗条件**: この判断が誤りだったと判明するのはどんな条件・兆候のときか
- **却下した代替案**: 検討した代替案(最低1つ)と却下理由

「失敗条件」は撤退基準としても機能する。運用開始後にその兆候が観測されたら
設計を見直すトリガーになる。

## 層3: 独立レビュー

成果物を確定する前に、**設計の経緯・会話の文脈を一切持たないレビュアー**に
設計書だけを渡してレビューさせる。設計者自身によるセルフレビューは、どれほど
誠実にやっても確証バイアスとアンカリングから逃れられないため、コンテキストの
分離そのものを対策とする。

Claude Codeで実行している場合は、Task(サブエージェント)ツールで汎用エージェントを
起動し、下記のテンプレートで指示する。

**渡すもの**(これだけ):

- 設計書本文(ファイルパスまたは全文)
- このスキルの `references/review-lens.md` と `references/antipatterns.md` のパス
- 出力形式として `assets/review-comment-template.md` のパス

**渡さないもの**(バイアス排除の要):

- 設計に至る会話の経緯・ユーザーとのやり取り
- 設計者による意図の説明・弁明・補足
- 「よくできていると思う」等の事前評価

## 独立レビュー指示文テンプレート

サブエージェントへの指示は次をベースにする。`{...}` を置き換える。

```text
あなたはデータ基盤アーキテクチャの独立レビュアーです。この設計の作成経緯は
知らされていません。それがこのレビューの意図です。設計書に書かれていない意図を
好意的に推測して補完せず、書かれている内容だけを評価してください。

対象: {設計書のファイルパス}

手順:
1. {review-lens.mdのパス} を読み、5柱+チームスキル適合性の全観点で設計書を評価する
2. {antipatterns.mdのパス} を読み、既知のアンチパターンに該当する箇所を検出する
3. 設計書の「主要設計判断」の各項目について、根拠が結論を支えているか、
   失敗条件が具体的か、却下された代替案の却下理由が妥当かを批判的に検証する
4. {review-comment-template.mdのパス} の形式で出力する。
   スコアカード、Severity順の指摘(Finding/Impact/Recommendation/Effort)、
   Strengths、総合判定を含めること

良い点(Strengths)も必ず挙げてください。ただし指摘を和らげるためではなく、
維持すべき決定を明確にするためです。
```

## 指摘への対応ルール

独立レビューの結果はそのまま採用するのではなく、次のルールで処理する。

1. すべての指摘に対して「対応」か「見送り」を決め、理由付きで設計書の
   「検証記録」セクションに記録する。無言で無視される指摘をゼロにする。
2. **CRITICALの指摘を未対応のまま成果物を確定しない。** 対応できない場合は
   その旨と理由を設計書に明記し、ユーザーに判断を委ねる。
3. 見送り判断の理由に「工数がかかるから」だけを書かない(サンクコストの隠れ蓑になる)。
   要件・制約に基づく理由を書く。
4. 指摘への対応で設計を大きく変えた場合は、変更部分について層1の照合をやり直す。

## サブエージェントが使えない環境でのフォールバック

Taskツールが使えない環境(一部のクライアント・API直接利用)では、次の
「コールドリビュー」で代替する。完全な文脈分離にはならないが、手順の分離で近似する。

1. 設計作業をいったん終了と宣言し、レビューフェーズに切り替える
2. 設計時の意図・経緯を参照せず、**設計書ファイルだけ**を読み直す
3. review-lens.md と antipatterns.md を(設計時に読んでいても)改めて読み、
   全項目を設計書の記述と1対1で照合する
4. 「設計書に書かれていないことは存在しない」ルールで評価する
   (頭の中の意図で行間を埋めない)
5. 出力・指摘への対応ルールは層3と同じ
