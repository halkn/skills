# GitHub Flow

feature branch → commit → PR → merge into main、という GitHub Flow を、ユーザーの短い一言から適切なステップに展開するためのスキル。ユーザーは毎回フルの手順を説明しないので、指示の文脈（直前の作業内容、リポジトリの状態）から今どのフェーズにいるかを判断して動くこと。

各フェーズは独立して頼まれることも、まとめて頼まれることもある。「commitして」の一言が実は「push して PR まで作ってほしい」を意味していることもあるので、指示を額面通りに取らず、直前の会話とリポジトリの状態（すでに push 済みか、PR が既にあるか等）から意図を推測し、不明なら一言で確認する。

## Phase 1: 状況把握

どのフェーズでも、まず現状を確認してから動く。

```bash
git status --short
git diff --stat
git log --oneline -10
```

- 作業ディレクトリに、今回の指示と無関係な未コミット変更がないか確認する。あれば触らない（ユーザーの作業中の変更を巻き込まない）。
- すでに feature ブランチにいるか、まだ main にいるかを確認する。

## Phase 2: ブランチ作成

まだ main（または base ブランチ）にいる場合、作業前に必ずブランチを切る。

- 変更内容からブランチ種別を判断し、prefix を付ける: `feat/`, `fix/`, `refactor/`, `docs/`, `chore/` など。プロジェクトに既存の命名規則があればそれに合わせる。
- `git switch -c <branch-name>` で作成する（`git checkout -b` は使わない — このリポジトリ群では `switch`/`restore` に統一されている）。

## Phase 3: コミット

- 対象ファイルだけを `git add` する。`git add -A` / `git add .` は使わない — 無関係なファイルの巻き込みを防ぐため。
- 1コミット1目的を守る。無関係な変更が混ざっていたら、コミットを分けることを提案する。
- コミットメッセージは小文字の conventional prefix（`fix:`, `add:`, `feat:`, `refactor:` など）+ 短い要約。必要なら箇条書きで詳細を続ける。プロジェクトの CLAUDE.md/AGENTS.md に別のコミット規約があればそちらを優先する。
- メッセージはヒアドキュメントで渡す（改行・クォートの事故を防ぐため）:

```bash
git commit -m "$(cat <<'EOF'
fix: 短い要約

- 詳細な変更点（必要な場合のみ）
EOF
)"
```

- Co-Authored-By 等の署名フッターは付けない。プロジェクトの attribution 設定に従う（明記がなければ付けないのがデフォルト）。

## Phase 4: PR 作成

- push 済みか確認する（`git log main..HEAD` で未 push のコミットがないか、リモート追跡ブランチの有無）。未 push なら `git push -u origin <branch-name>`。
- `gh pr create` で以下の形式の本文を書く（日本語ユーザーには日本語で。プロジェクトの言語規約があればそれに従う）:

```markdown
## Summary
- 変更点を1〜3個の箇条書きで

## Test plan
- [ ] 実行した確認手順をチェックボックスで
```

- タイトルは変更内容が一目で分かる短い一文にする。

## Phase 5: マージ

「マージして」と言われたら:

- `gh pr checks <num>` で CI の状態を確認する。失敗があれば実行せず、ユーザーに報告する。
- 通常マージする: `gh pr merge <num> --merge`（squash や rebase ではなく、履歴を保持する通常の merge をデフォルトとする）。プロジェクトに別のマージ方針の指定があればそちらに従う。

## リスクの高い操作への注意

`git push --force`、`git reset --hard`、ブランチ削除などの破壊的・共有影響操作は、このスキルの範囲外の判断が必要になるので、実行前に必ずユーザーに確認する。
